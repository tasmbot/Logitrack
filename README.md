# Logitrack

## Глава 1. Введение.
Цель курсовой работы — повышение эффективности и качества доставки грузов за счет проектирования системы управления данными.
Для достижения поставленной цели необходимо решить следующие задачи:
*	провести анализ предметной области и существующих решений в сфере управления данными в логистике;
*	определить функциональные и нефункциональные требования к будущей системе;
*	разработать архитектуру проектируемого решения, включая описание бизнес-процессов и диаграммы UML;
*	спроектировать логическую и физическую модели базы данных;
*	реализовать элементы управления данными (триггеры, функции, процедуры), обеспечивающие корректность и целостность информации;
*	создать пользовательский интерфейс для работы с системой;
провести оценку соответствия разработанной системы исходным требованиям и сформулировать выводы.

Объект исследования — автоматизированная информационная система, предназначенная для управления данными о грузах и процессами их доставки.

Предмет исследования — процессы сбора, обработки, хранения и предоставления информации о движении грузов и маршрутах доставки, включая:
*	базу данных для хранения информации о заказах, клиентах, маршрутах и транспортных средствах;
*	алгоритмы контроля и анализа перемещений грузов;
*	пользовательский интерфейс для сотрудников компании;
механизмы визуализации данных и формирования отчетности.
Методы исследования, используемые в работе, включают:
*	анализ предметной области и обзор существующих решений;
*	моделирование бизнес-процессов с применением нотации BPMN;
*	объектно-ориентированное проектирование с использованием UML-диаграмм;
*	проектирование базы данных с использованием ER-моделирования и нормализации;
*	реализацию SQL-компонентов (триггеров, функций, процедур);
*	разработку графического интерфейса средствами языка программирования Python.

## Глава 2. Требования к проектируемой системе.
Функциональные требования определяют перечень задач, которые должна решать система. Перечислим функциональные требования для рассматриваемой системы:
*	регистрация и авторизация пользователей с разграничением прав доступа для ролей: оператор, администратор, водитель, клиент;
*	ведение базы данных заказов и грузов с хранением информации о клиентах, характеристиках грузов (вес, объем, ценность), маршрутах и статусах доставки;
*	поддержка регистрации и редактирования заказов несколькими операторами одновременно без потери или дублирования данных;
*	мониторинг местоположения грузов и транспортных средств в реальном времени с обновлением информации не реже 1 минуты;
*	управление заказами для операторов;
*	формирование отчетов по доставкам (общее количество заказов, выручка, средний вес);
*	визуализация данных через графики: загрузка транспортных средств, динамика заказов, статистика по клиентам;
*	поддержка уведомлений для клиентов о статусе заказа (создан, в пути, доставлен) через приложение;
*	контроль целостности данных и ведение журнала действий пользователей (логирование).

Нефункциональные требования определяют качественные характеристики системы. Для рассматриваемой системы управления доставкой грузов они могут быть сформулированы следующим образом:
*	время обновления информации о местоположении грузов и состоянии доставки не более 1 минуты;
*	обеспечение многопользовательского доступа с разграничением прав для ролей: оператор, администратор, водитель, клиент;
*	авторизация пользователей по паролю с защитой от несанкционированного доступа;
*	возможность резервного копирования базы данных не реже 1 раза в день и восстановления системы из резервной копии;
*	единый стиль интерфейса с интуитивно понятной навигацией и наглядной визуализацией маршрутов и статусов грузов;
*	интерфейс на русском языке.

Технические требования определяют архитектурные ограничения и условия интеграции системы управления доставкой грузов. Система должна быть реализована с использованием реляционной СУБД, поддерживающей обработку транзакций, целостность и консистентность данных. Предусматривается возможность интеграции с внешними системами, включая банковские сервисы, GPS-трекеры и корпоративные информационные платформы, через стандартизированные API-интерфейсы.

Эксплуатационные требования включают обеспечение круглосуточной технической поддержки с регламентированным временем реакции на инциденты.

Требования к защите информации предусматривают многоуровневую систему безопасности. Система должна функционировать в многопользовательском режиме с разграничением прав доступа по ролям (оператор, диспетчер, курьер, клиент, администратор) с целью защиты от несанкционированного вмешательства и обеспечения конфиденциальности данных клиентов и информации о грузах. Резервное копирование данных должно выполняться ежедневно.
